# Backend 设计原则

> 2026-06-10 立。决定 agent-bridge 怎么接第二个 backend（hermes_acp），
> 同时把"为什么 backend 长成现在这样"沉淀下来，避免下一个 backend 重新争论。

读这份之前，先读 `架构.md` 了解 channel/router/文件协议三层。

## 一句话

**Backend 是"消费 inbox.txt → 产出 outbox.txt"的黑盒**。怎么消费、怎么产出，是
backend 自己的实现细节。router、channel、web 都不关心。

## 文件协议是 backend 唯一的契约

router 写 `chat_sessions/<alias>/inbox.txt`，web/channel 读
`chat_sessions/<alias>/outbox.txt`。这两个文件 + `meta.json` + `history.json`
是 backend 跟 agent-bridge 其它部分通信的**全部接口**。

| 文件 | router 视角 | backend 视角 |
|---|---|---|
| `inbox.txt` | 写：用户消息 | 读：拿到要处理的消息 |
| `outbox.txt` | 读：拿回复推给用户 | 写：覆写最新的"待推送回复" |
| `meta.json` | 读：`daemon_pid` 判活、`backend` 字段选 daemon 模块 | 写：alias / cwd / 各种 pid / created_at |
| `history.json` | 写：每轮 user / assistant 追加 | 不读不写 |
| `spawned_pids.jsonl` | 不读 | 写：daemon 自己 spawn 的子进程 PID + create_time |

**协议特性**（落 backend 前必读）：

- `outbox.txt` 是**覆写**，不是追加 —— "最新一条待推送的回复"，watcher 用
  `[stamp]|reply[:120]` 当指纹去重。要支持流式回复必须升协议（见后文）。
- `inbox.txt` 同样是覆写，由 router 在新消息到来时写入；backend 读完后不要清空
  —— 让 watcher 用 mtime 判新。
- `meta.json` 必填字段：`alias`、`cwd`、`daemon_pid`、`child_pid`、`created_at`、
  `backend`（新增）。dead 之后追加 `last_exit_at` 并把两个 pid 清成 null。
  从 2026-06-11 起 init_lifecycle 还写 `daemon_create_time`
  （`psutil.Process.create_time()` 浮点秒），用于 PID 复用防护——老会话没
  这个字段时 lazy-fix 退化到只看 PID 活性，兼容。
- `last_exit_at` 三种来源：(a) daemon 自己 atexit 钩子写的 ISO 时间戳——正常退出；
  (b) `"(detected_dead)"`——`core.sessions.list_sessions` 扫描时发现 meta 字面
  声称在线但 PID 已不活时的 lazy-fix 标记（被 `taskkill /F` / OOM / 解释器崩等
  绕过 atexit 时会落这种）；(c) `"(detected_dead_pid_recycled)"`——同 (b)，
  但是 PID 还在、`psutil.Process.create_time()` 对不上 meta 里记的，说明
  那是 OS 复用后的另一个进程。读 meta 的人不需要区分这三种，只关心"有这
  字段 = 已离线"。

## daemon 在消息路径上吗？由 backend 决定

这是设计 backend 时最关键的一个判断，**没有"对"的答案**，取决于下游 agent 的协议形态：

### 路径外（claude_code 是这种）

```
router → inbox.txt
             ↑ poll
        ┌────┴────┐
        │ child claude（持有 mcp_bridge）  ← daemon spawn 它，但不管消息
        └────┬────┘
             ↓ write
         outbox.txt → web/channel
```

- daemon 只负责"让 child claude 一直活着且处于 wait_for_message 状态"
- 消息流转完全发生在 daemon 之外（child claude 内嵌的 mcp_bridge 自己 IO 文件）
- daemon 崩了 child claude 还能继续干活；下次 inbound 时 router 会 respawn 一个 daemon 接管看护

### 路径内（hermes_acp 是这种）

```
router → inbox.txt
             ↑ poll
        ┌────┴────┐
        │ daemon (持有 ACP stdio socket)
        └────┬────┘
             │ session/prompt → ACP → session/update chunks
             ↓ aggregate + write
         outbox.txt → web/channel
```

- daemon 自己 poll inbox、跟下游 agent 走 stdio JSON-RPC、聚合事件、写 outbox
- 没有"内嵌一个像 mcp_bridge 一样的子组件"的概念——下游 hermes 自己不读文件
- daemon 崩了消息流转立刻断

### 选哪种由"下游 agent 怎么收消息"决定，不由 agent-bridge 喜好决定

| 下游 agent 提供的接口 | daemon 站位 | 例子 |
|---|---|---|
| 在自己进程内可以 host 我们的代码（MCP server / 插件 / SDK） | 路径外 | claude_code（mcp_bridge 跑在 claude 进程内） |
| 只暴露外部 RPC / API / stdio | 路径内 | hermes_acp（ACP 是 stdio JSON-RPC，agent-bridge 必须在外面持有 socket） |

不要为了"统一"而强行让两个 backend 走同一路径。下游协议是什么样、daemon 就长什么样。

## 为什么不走"统一 daemon 模型"（B 方案）

讨论过一个方案：让 daemon 统一变成消息泵，所有 backend 都走
`router → daemon → BackendDriver.send/wait_for_reply → outbox`。**实测下来这是
过度抽象**，没有解锁任何具体能力：

| 假想优势 | 真实情况 |
|---|---|
| 可观察性集中在 daemon | 应该埋在 channel + router（含 inbound→outbound 全链路时间戳），跟 backend 内部架构无关 |
| 撤回/中断统一接口 | A 也能加（`inbox.cancel` 文件 + 各 backend 自己实现 cancel handler），代码量相当 |
| 流式回复 | 瓶颈是 outbox 协议升级（覆写 → jsonl tail），跟 daemon 站位无关 |
| 多用户路由 | 这是 router 层的事（peer_id → alias 映射），跟 backend 内部无关 |
| 统一 usage 上报 | 写一个 `usage.jsonl` 就够，两个 backend 各往里写 |
| daemon 是路径唯一节点 | 审美，没解锁能力。代价是 claude_code 链路被迫拆 mcp_bridge，把"消息能力 = child claude 活着"这个天然不变量打破 |

B 方案唯一真实价值：**daemon 做横向决策**（基于历史限流、跨 session 协调、缓存、去重、重试）。
但这些事根本不是 backend daemon 的视野，是 router / core 的事 —— daemon 只看一个 alias 一个下游 agent，没有跨 session 视角，也没必要有。

**结论**：A 方案（各 backend 选自己的 daemon 站位）是各司其职，B 方案是用复杂度换审美统一感。走 A。

## daemon 职责拆分

观察 `backends/claude_code/daemon.py` 现有职责，能区分出**通用**和**backend 专属**两类：

### 通用（抽到 `core/daemon_lifecycle.py`）

- CLI 解析 `<alias> <cwd>`，alias 缺省时用 `make_alias_for_cwd(cwd)` 生成
- alias 校验 + `chat_sessions/<alias>/` 目录创建
- 决定 spawn cwd：CLI > meta.json 历史 > backend 默认 > $HOME 兜底
- 日志：`daemon.log` via `logging.basicConfig`
- `meta.json` 原子写：alias / cwd / daemon_pid / child_pid / created_at / backend / 自定义 extra
- `spawned_pids.jsonl` 追加（含 psutil create_time 防 PID 复用）
- `atexit` + `SIGINT` 清理：调 backend 的 on_exit 回调，最后改 meta 写 last_exit_at + 两个 pid 清 null

### Backend 专属（留在 `backends/<name>/daemon.py`）

- 怎么 spawn 下游进程（claude_code：pywinpty + `claude.exe`；hermes_acp：subprocess + `hermes acp`）
- 怎么知道下游 ready（claude_code：等 TUI 欢迎屏 marker；hermes_acp：initialize / new_session response）
- 怎么处理下游异常（claude_code：trust-folder 对话框、rate-limit 撞墙后 press "3"；hermes_acp：session 失效后 reinit）
- 消息流转（如果 daemon 在消息路径上才做）

把这两类分开之后，新 backend 的 daemon 就是 ~50-100 行胶水加上必要的下游适配，
而不是从零抄一遍 ~330 行的 claude_code/daemon.py。

## 跨 backend 能力归属表

下面几个能力将来都可能要做，先把"该在哪一层做"钉死，避免下次又在 daemon 抽象上纠结：

| 能力 | 该在哪一层做 | 为什么 |
|---|---|---|
| 撤回 / 中断 turn | backend 内（各 daemon 实现 cancel handler，约定 `inbox.cancel` 文件触发） | 撤回必须能调下游具体的中断 API，跨 backend 的下游协议不一样 |
| 流式回复 | outbox 协议升级（覆写 → `outbox.jsonl` tail）+ 两个写端各支持 | 这是文件协议本身的演进，跟 daemon 站位无关 |
| 多用户路由（peer_id → alias） | router | router 是入口，是唯一拥有"哪个 peer 该映射到哪个 alias"全局视角的地方 |
| 限流（按 alias 历史 / 跨 session） | router | router 在消息进系统的第一道闸，有 history.json 全视角 |
| 去重 | router | 同上 |
| 重试 | 看重试什么。下游 API 调用失败 → backend 自己重试；消息投递失败 → channel 重试 | 跨边界的重试不应该跨层 |
| 缓存 | router 或 core | 基于 inbox 内容做命中判断，daemon 拿到时已经晚了一层 |
| Usage 上报 | backend 写 `usage.jsonl`，core 聚合 | 数据源在 backend，聚合视角在 core |
| 跨 session 协调 | core | daemon 没有跨 session 视角，强行让它有就是 B 方案的坑 |

**判断准则**：决策需要的视角在哪一层，就在哪一层做。daemon 视角 = 一个 alias 一个下游 agent。
凡是需要多 session 视角、需要 history 视角、需要 peer 视角的，都不该归 daemon。

## 添加新 backend 的清单

详细步骤参见 `docs/新增后端.md`。这里只列结构决策：

1. 看下游 agent 的协议形态 → 决定 daemon 站位（路径外 / 路径内）
2. `backends/<name>/daemon.py`：用 `core/daemon_lifecycle` 起骨架，加下游适配
3. （路径外）一般还要 `<name>/bridge.py` 或类似——下游进程内运行、IO 文件
4. （路径内）daemon 内部加 inbox watcher + 下游协议 client + outbox writer
5. `meta.json` 写 `backend: "<name>"`
6. `core/spawn.py` 已经按 `meta.backend` 字段选 daemon 模块（默认 `claude_channel`）
7. 创建会话时显式指定 backend（CLI / web /new 入口要带这个参数）

| 维度 | ~~claude_code（已删）~~ | hermes_acp | claude_channel（默认） |

| 启动开销 | ~10s（TUI 加载 + 触发 chats-loop） | 首次 ~60-80s，后续 ~1s | ~10s（TUI 加载 + 确认 dev-channel 警告框） |

## claude_channel backend 契约（2026-07-20 设计，代码未落）

> 目标：用 Claude Code channels（研究预览）替代 chats-loop 轮询。预研核心闭环
> 已验证通过，见 `docs/CHANNELS预研.md`。本节是落 backend 前的契约。

### 定位

**claude_channel = mcp_bridge 的"推模型"版。** 和 claude_code 几乎同构（都是路径外、
都在 claude 进程内 host 我们的组件、都遵守 inbox/outbox 文件协议），唯一本质差异：
消息到达后不再靠 `wait_for_message` 拉给 claude，而是靠 MCP channel notification
**主动推**进会话。这消掉了 chats-loop 循环、marker 心跳租约、ESC 重唤那整套 pull
模型补丁。

### daemon 站位：路径外

按判据表"下游 agent 能在自己进程内 host 我们的代码"→ 路径外。channel_server 是
claude 的 stdio 子进程，notification/reply 直接进出会话，daemon 只看护 claude 进程活着。

### 进程拓扑（含 backend 内部 daemon↔channel_server 分工）

```
daemon.py (Python，winpty 拉 claude，**不带 --strict-mcp-config**，带 dev-channel flag)
  │  ①poll inbox.txt (mtime+baseline) ──localhost POST──▶ channel_server /inject
  │  ③写 outbox.txt  ◀──localhost POST── channel_server (reply 到达时回调)
  └─ PtyProcess: claude.exe  ← --dangerously-load-development-channels server:wxchan
       └─ stdio 子进程: channel_server.mjs (JS/MCP SDK，极薄，不碰文件)
            │  ②收 /inject → notification 推进会话
            └─ claude 调 reply 工具 → POST 回 daemon 回调口（③）
```

**关键分工（2026-07-20 实现层细化，daemon 管文件、channel_server 只管协议）**：
文件 IO 与路径/格式逻辑全留在 **Python daemon**（复用现有 `inbox_path` /
`outbox_path` / `_write_outbox` helpers），channel_server 是**纯协议转换器**，
彻底不碰文件系统——避免 JS 端复刻 ROOT/paths 计算（本项目反复踩的坑，见 CLAUDE.md）。

### 入站契约：文件协议不变（同 hermes / claude_code）

router → inbox.txt 这条**对外契约完全不变**。变的只是"谁在 backend 内部读
inbox"：由 **daemon（Python）** poll inbox.txt（同 hermes `_drain_inbox`，
mtime 判新 + startup baseline 防重放），拿到新消息后 **localhost POST** 到
channel_server 的 `/inject`。

> ⚠️ 这里的 HTTP 是 **daemon↔channel_server 的进程内 localhost 通信**，端口只在
> 本 backend 内部用、随 claude 进程生灭，**不是**契约里否掉的"router HTTP 直投"。
> router 仍只写 inbox.txt，不知道任何端口。二者别混淆。

- **防重放**：daemon startup 以当前 inbox mtime 为 baseline，只处理之后的写入。
- **outbox**：claude 调 reply → channel_server POST 回 daemon 回调口 → **daemon**
  覆写 outbox.txt，复用 `_write_outbox`（格式 `[HH:MM:SS]\n<reply>\n`，同 mcp_bridge）。

### 启动序列（backend 专属，daemon 负责）

1. winpty spawn `claude --mcp-config <配置> --dangerously-load-development-channels
   server:wxchan --dangerously-skip-permissions`（**死记：不带 `--strict-mcp-config`**，
   它会屏蔽 dev channel 注册——见 CHANNELS预研.md 卡点根因）。
2. spawn 后 PTY 自动喂 `\r` 确认 dev-channel 全屏警告框（默认选项 1 = local
   development）。复用 spike_tui.py 阶段1 逻辑。
3. ready 判定：channel `/health` OK（若产品版保留 HTTP 探活口）或 TUI 欢迎屏
   marker。写 ready marker 让 web spawn.watch_ready 发就绪通知。

### 可服务性判定：**claude 进程活 = 通道活**

channel_server 是 claude 的 stdio 子进程，同生命周期。所以在线判定退化到最简单的
"claude 进程 PID 活"——**marker 心跳租约整套退役**，这是 claude_channel 相对
chats-loop 的最大架构收益。claude_code 那套 `daemon 活 || (bridge 活 && marker 新鲜)`
的复杂判定在这里不需要。

### 权限：首版 skip-permissions 托管

首版用 `--dangerously-skip-permissions`，和现有 claude_code daemon 一致，先把主
链路跑通。权限中继（permission_request → 微信回 `yes <id>` → verdict，见
CHANNELS预研.md #6）留后续 spike + 迭代。

### channel_server 实现要点

- 从 spike 的 `wxchan.mjs` 产品化，落 `backends/claude_channel/channel_server.mjs`。
  **这是本项目第一个带 .mjs / node 依赖的 backend**（MCP channel 官方路径是 JS
  SDK；Python 侧发 channel notification 未验证，不冒险）。
- 相比 spike 版要加：inbox poll + baseline 防重放、outbox 覆写、**sender allowlist**
  （CHANNELS预研.md 安全节：无门禁通道 = 提示注入入口，必须按 sender id 过滤，
  不在名单静默丢）。
- channel_server 要薄，channel 协议细节集中在这一处，便于预览期契约变更时集中改。
- **instructions 必须强制 claude 用 reply 工具回复**（2026-07-20 e2e 实测坑）：
  channel 的 `instructions`（进 claude system prompt）要明确告诉 claude"用户不在
  终端、看不到你打印的任何文本，回消息的唯一途径是调 reply 工具，每条 channel
  消息都必须以调 reply 收尾"。否则 claude 会把回复当普通 TUI 文本输出，reply
  工具没被调用 → outbox 永远空 → IM 用户永远收不到。这不是可选优化，是产品必需。

### resume 控制通路（2026-07-21 设计，代码未落）

**目标**：让 claude_channel 会话能"接回上次的上下文"，而不是每次白纸重开。
Claude Code 原生 `--resume <session-id>` 支持这个，探路已实测 **`--resume` 与
`--dangerously-load-development-channels` 完全兼容**（传确切 session-id 时不弹
"选历史会话"选择器，启动序列仍是"弹 dev-channel 警告框 → 喂 `\r` → 就绪"，只是多带回
历史 transcript）。缺口只在"谁在运行期把 session-id 喂给正在跑的 daemon"。

**通路 = 复用 `pty_control.txt`，加 `RESUME:` 前缀（方案 A）**。不新开文件/端口：

1. router 选中会话后，把 `RESUME:<session-id>` 写进该 alias 的 `control_path`
   （`pty_control.txt`）。这文件和 per-alias 路径已存在（claude_code 用它回灌 PTY
   控制序列）。
2. claude_channel daemon 的 inbox poll 主循环**每轮顺便读一眼 `control_path`**：
   - 内容以 `RESUME:` 开头 → 解析出 session-id，执行 resume 重启（下面三步），
     读完即删该文件（one-shot，不残留）。
   - 其它内容 → claude_channel **不做 PTY 控制**（它不是 chats-loop），忽略即可。
3. resume 重启三步（daemon 内）：
   `kill 当前 child claude` → `带 --resume <id> 重新 spawn`（其余参数、env、喂 `\r`
   确认警告框的启动序列**完全复用** `main()` 里那段，抽成可带 resume 参数的函数）
   → `重走 ready 判定 + 写 ready marker`，就绪后经既有"就绪通知"链路告诉用户"已接回"。

**为什么走 A 不新开 `resume_request.txt`（方案 B）**：跟"消息靠文件 + mtime poll"的
全系统风格一致，daemon 主循环只多一个"读一眼 control 文件、认 `RESUME:` 前缀"的分支，
不引入新基建。B 语义更隔离但要 router + daemon 各多认一套文件约定，净造一套东西。

**边界**：
- session-id 由 router 从磁盘 jsonl 现扫现给（见 `入站路由.md` "resume 菜单"），
  daemon 不猜、不记 alias→session 映射。传进来找不到对应 transcript → `--resume`
  会失败，daemon 要 catch 并回落"开新会话 + 提示 resume 失败"，不能崩。
- kill 旧 child 是**有意的顶替**（符合"全局单选中会话"模型）：一个 alias 同一时刻只
  有一个活 child，resume = 换掉它。孤儿回收沿用现有 spawned_pids 清理逻辑。
- 只有 claude_channel 认 `RESUME:` 前缀。claude_code 的 `control_path` 语义不变
  （它写的是纯数字 PTY 控制序列，永远不会以 `RESUME:` 开头，两者不冲突）。

**resume 失败自愈（2026-07-23，实测踩坑后加）**：`_do_resume` 里 `--resume` 重 spawn
后 `_spawn_and_wait_ready` 返回 `(new_proc, ok2)`。`ok2=False`（channel_server never
healthy / dev-channel 警告框没确认）时**决不能留着那个 new_proc**——它进程活着但
channel 死了，会骗过主循环的 `proc.isalive()` 检测，daemon 空转、inject 一直失败、
却仍显示在线，消息落死 inbox 没反应（这就是"显示在线实际不在线"的机制）。所以失败处理：

1. **kill 掉没就绪的 new_proc**（不留"进程活但 channel 死"的僵尸 child）。
2. **清 `meta.child_pid`**（→ online 判据 `child 活` 变 False，会话正确显示离线）。
3. **写 outbox 提示 + daemon 退出**（`return`）。退出后该会话 online=False。
4. **不自动重试**：用户下条消息 → 无活 daemon → 走 idle/零会话 → 弹 /proj → 用户
   重新选会话 → 重新 resume。重试与否交给用户（多次失败用户自会察觉这会话有问题），
   daemon 不闷头重试、不自作主张替用户重选。与"不许手动替用户做会话选择"原则一致。

### STOP 控制信号：daemon 主动收尾（2026-07-23）

**终结一个会话必须让 daemon 自己收尾，不能从外面 psutil 硬杀。** daemon 是管理者，
child 意外崩了它会重启/报错（看护语义）；但"用户主动要求终结"是**另一回事**——
从外面只杀 child，daemon 分不清是崩了还是你想停，会重拉或报错；只杀 daemon 又留 child
孤儿。所以给 daemon 一个**主动终结信号**，让它自己杀 child + 退出，清干净。

- **通路**：复用 `control_path`（跟 `RESUME:` 同机制）。router 写 `STOP`（无参数）。
- **入口**：微信 `/kill`（无参数，终结 current 会话；发一次即杀，不二次确认）→
  `_cmd_kill` 往 current 的 `control_path` 写 `STOP`。
- **daemon 主循环**：`_check_resume_signal`（读 control_path）扩展认 `STOP` →
  **主动 kill 自己的 child + 清 meta（daemon_pid/child_pid 置 None）+ 干净退出**
  （`return 0`）。**关键**：这是"你要它停"，不是 child 崩了，所以不走"进程已退出、
  请 /new 重开"那条报错分支——置一个"主动终结"标志让主循环干净 return。
- **退出后** online=False（daemon 死+child 死），会话壳保留（transcript 也在），
  下次 `/proj` 仍能 resume 接回。要连壳一起清是"3 天 GC"或另外的删数据操作，STOP 只停进程。
- 只有 claude_channel 认 `STOP`。与 `RESUME:` 一样，claude_code 已删不涉及。

claude_channel 是**并列的第三个 backend**，不动 claude_code / hermes_acp。验证成熟
前 chats-loop（claude_code）保持默认，claude_channel 通过 `/backend claude_channel`
或 dashboard 显式选。等 claude_channel 稳了再考虑是否切默认、是否退役 chats-loop 补丁。

## 不要做的事

- 不要把"通用 daemon 模型"再翻出来——已经讨论过两遍，不解锁任何能力（B 方案）
- 不要给 daemon 加跨 session 视角的功能——那是 core/router 的事
- 不要为了"backend 形态对称"硬改 claude_code 链路——mcp_bridge 现状是经过实际场景打磨的，不要为了审美回归风险
- 不要在 backend 层做用户路由——router 一处够了

## 默认 backend 契约（命令行入口用）

命令行入口（微信 `/proj`、`/new`）建会话时不让用户在每条消息里选 backend——
手机微信塞个选择 UI 体验会乱。改用一个**粘性的"默认 backend"**：

- 文件：`chat_sessions/_default_backend.txt`，内容是 backend 名（`claude_code` /
  `hermes_acp`），缺省（文件不存在或值不在 `KNOWN_BACKENDS` 里）按 `claude_code`。
- 读：`core.sessions.get_default_backend()`。
- 写：`core.sessions.set_default_backend(name)` 或 `/backend <name>` 命令。
- 作用域：只影响**之后**通过 `/proj N` / `/proj 0` 新建的会话；已有会话不变。
- web dashboard 建会话走自己的 backend 参数（modal 下拉），不读这个文件——
  浏览器有空间显式选，命令行没有。

### 已知 backend 集合

`core.sessions.KNOWN_BACKENDS` 是单一来源（目前 `("claude_code", "hermes_acp")`）。
新加 backend 时同步：
1. 这里加一项
2. `core.spawn._BACKEND_DAEMON_MODULES` 注册一条
3. `web.spawn_helpers._KNOWN_BACKENDS` 同步（dashboard 校验用）

为什么不让 `core.sessions` 反查 spawn 的注册表：sessions 是底层，spawn 依赖
sessions（加载 meta），方向反过来会成环。两处同步成本可接受，未来若变成 3+ 个
backend 再考虑抽到 paths.py 之类的更底层。

- **2026-06-10**：决定 hermes_acp 走"1 alias = 1 daemon = 1 hermes acp 子进程"，
  跟 claude_code 同构。ACP 协议本身支持"单进程多 session"（让 N 个 alias 共享一个

  - 失败域隔离更强：一个 alias 的 hermes 崩了不影响其它 alias
