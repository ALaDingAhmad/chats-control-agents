# agent-bridge 架构

把"IM 渠道"（人在打字的地方）和"AI 执行后端"（在回话的东西）解耦的桥。
最初是把本地 Claude Code TUI 通过微信暴露给手机；现在抽象成插件化——
加飞书或 OpenClaw 就是 `channels/` 或 `backends/` 下加一个文件。

## 数据流

```
┌─────────┐  入站文本       ┌──────────┐    route_inbound(text, src)   ┌─────────────┐
│ 微信     │ ───────────────▶│ channels │ ─────────────────────────────▶│ core.router │
│ 浏览器   │                 │ （只 I/O │                                │ • slash 命令│
│ 飞书?   │ ◀────────────── │  不路由）│ ◀───────────────────────────── │ • idle gate │
└─────────┘  回复 / typing   │          │   RouteOutcome(reply|routed)   │ • 拉 daemon │
                            └──────────┘                                │ • 写 inbox  │
                                                                        └──────┬──────┘
                                                                               │
                                                                               ▼
                                                                     ┌─────────────────┐
                                                                     │ chat_sessions/  │
                                                                     │   <alias>/      │
                                                                     │     inbox.txt   │ ◀── mcp_bridge 轮询
                                                                     │     outbox.txt  │ ──▶ web /poll + outbox_watcher 轮询
                                                                     │     history.json│
                                                                     │     meta.json   │
                                                                     └─────────────────┘
                                                                               ▲
                                                                               │ inbox 轮询 / outbox 写
                                                                               │
                                                                     ┌─────────────────┐
                                                                     │ 后端（按 meta   │  claude_code：daemon spawn child
                                                                     │ .backend 选）   │    claude.exe，带 CHATS_LOOP_ALIAS，
                                                                     │ • claude_code   │    child 内嵌 mcp_bridge 自己 IO 文件
                                                                     │ • hermes_acp    │  hermes_acp：daemon 自持 ACP stdio，
                                                                     └─────────────────┘    自己 poll inbox / 写 outbox
```

## 会话模型

一个 *session* 是 `chat_sessions/<alias>/` 目录。内含文件：

| 文件 | 谁写 | 谁读 |
|------|-----|-----|
| `inbox.txt`          | 渠道入站路径（web /send、weixin longpoll）通过 router | 后端（mcp_bridge `wait_for_message` 或 hermes daemon poll） |
| `outbox.txt`         | 后端 `send_chat_response` / daemon 就绪通知          | web /poll + weixin outbox watcher    |
| `history.json`       | router/web routes（每轮追加，含 pre_send_hook 采收的终端旁白） | UI                          |
| `meta.json`          | daemon（start/exit，经 daemon_lifecycle）+ 命令（`/new`）+ mcp_bridge（bridge_pid 注册） | sessions.list_sessions、projects.py、router 可服务性判定 |
| `spawned_pids.jsonl` | daemon（每次 spawn 追加）                           | pid_track.list_daemon_child_pids    |
| `pty_control.txt` `pty_control_mode.txt` | router（数字控制序列）/ daemon（菜单 arm） | daemon / router（one-shot 消费） |
| `daemon.log` `pty.log` `daemon_stdout.log` | daemon                          | 人看                                 |

### 会话两种形态

| | daemon-managed | bridge-owned（终端 chats-loop） |
|---|---|---|
| 怎么来的 | `/proj` 选号 / dashboard 新建 → autospawn daemon | 用户手开 claude 窗口跑 `/chats-loop` |
| 谁在收件 | daemon（或其 child claude 的 mcp_bridge） | 该窗口的 mcp_bridge，无 daemon |
| 在线判据 | `daemon_pid` 活 | `bridge_pid` 活 **且** marker 新鲜（`loop_marker_fresh`，TTL 180s 心跳租约） |
| 死了怎么办 | router `ensure_daemon_alive` 自动复活 | 不叠 daemon（防双读 inbox），回 `/proj` 菜单让用户选 |

完整契约见 [`docs/入站路由.md`](入站路由.md) "终端 chats-loop 会话（bridge-owned）"节。

## 当前选中 alias

`chat_sessions/_current.txt` 存一个 alias。所有渠道入站的普通消息都路由
到**这个** alias 的 inbox。**只有用户显式选择能改它**（`/use`、`/proj` 选号、
dashboard、菜单 one-shot 选号）——mcp_bridge/daemon 等基础设施绝不写它，
否则随手开个 claude 窗口就会劫持全部入站路由。单一全局选中 ——
agent-bridge 当前是单用户场景，多用户路由要按 `peer_id` 分流，暂未实现。

## /proj 数字选择流程

`/proj` 输出编号列表后，用户回纯整数即可选中。**one-shot 语义**：菜单只在
"刚弹出的下一条入站"认数字，任何非数字入站即解除 arm（详见
[`docs/入站路由.md`](入站路由.md) "纯数字入站"节）。状态（列表 + 兜底 TTL）
持久化到 `chat_sessions/_pending_proj.json`，扛 web_server 重启。

特殊值：`0` 表示开空会话（cwd=用户主目录、不绑项目），见 `commands._cmd_pick_proj`。

## Autospawn

`/proj` 选中离线 / 不存在的项目时，`core.autospawn.request_autospawn(alias, cwd)`
追加到 `chat_sessions/_autospawn_queue.jsonl`。web_server 的
`web.autospawn.autospawn_worker` 后台任务 drain 队列，detached 方式 spawn
`python -m <daemon 模块> <alias> <cwd>`——daemon 模块由
`core.spawn._resolve_daemon_module` 按 `meta.json.backend` 决定
（`claude_code` 或 `hermes_acp`；Windows:
`CREATE_NEW_PROCESS_GROUP | CREATE_NO_WINDOW`）。新建会话的 backend 来源：
命令行入口读 `chat_sessions/_default_backend.txt`（`/backend` 命令改），
dashboard 用 modal 下拉选。

## 渠道 ↔ router ↔ 后端 交互

渠道调 `core.router.route_inbound(text, source)`，按返回的 `RouteOutcome`
分流 —— 渠道**永远不直接调** `commands` / `sessions` / `spawn`。router 负责
slash 命令、passthrough、one-shot 菜单消费、2h idle gate、no-session 兜底、
执行端可服务性（死 daemon 复活；bridge-owned 会话循环没跑就回菜单）、写
inbox / history。完整契约：[`docs/入站路由.md`](入站路由.md)。

bridge 主动拉起 daemon 之后（router 的 ensure_daemon_alive 或 autospawn
worker），会启动 `spawn.watch_ready(alias)` 后台任务监听
`~/.claude/.chats-loop-active-<alias>` marker，就绪/失败时写 outbox 通知
用户——所有渠道通过现有 outbox_watcher 自动收到。完整契约见
[`docs/入站路由.md`](入站路由.md) "就绪通知"节。

## claude_code daemon 生命周期

child claude 怎么拉起（trust-folder 对话框、ready 标记、trigger、drain
循环）、rate-limit 看门狗、怎么加新弹窗处理器（工具授权、操作确认）——
详见 [`docs/daemon生命周期.md`](daemon生命周期.md)。

## Backend 设计：daemon 在不在消息路径上由 backend 决定

`claude_code` 是"daemon 路径外"型：daemon 只看护 child claude，消息流转由
child claude 内嵌的 mcp_bridge 自己 IO 文件。daemon 死了消息不断。

`hermes_acp`（已接入）是"daemon 路径内"型：daemon 自己持有 ACP stdio
socket，自己 poll inbox 写 outbox。daemon 死了消息立刻断。

两个 daemon 共用 `core.daemon_lifecycle` 骨架（日志/meta/atexit/
resolve_spawn_cwd——cwd 无效时抛错并写 outbox 通知，不再静默兜底）。
新加 backend 要同步 3 处注册：`core.sessions.KNOWN_BACKENDS`、
`core.spawn._BACKEND_DAEMON_MODULES`、`web.spawn_helpers._KNOWN_BACKENDS`。

两种站位都合法 —— 取决于下游 agent 的协议形态（能不能在它进程内 host 我们
的代码）。**不要为了"统一"硬改 claude_code 链路**。详见
[`docs/后端设计.md`](后端设计.md)。

跨 backend 的能力（撤回 / 流式 / 多用户路由 / 限流 / usage 上报）该在哪
一层做，那份文档里有归属表。简短规则：决策需要的视角在哪一层就在哪一层
做，daemon 视角 = 一个 alias 一个下游 agent，所以多 session / 多 peer 类
的决策都在 core / router 层。
