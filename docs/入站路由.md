# 入站路由契约

任何渠道（今天是微信，未来可能是飞书/Slack/…）收到用户消息后，统一交给
`core.router` 决策，渠道自己不直接调 `commands` / `sessions` / `history` /
`spawn`——这些是核心层的事。

## 入口

```python
from chats_control_agents.core import router

outcome: router.RouteOutcome = await router.route_inbound(
    text="...",                       # 用户原文，渠道收到啥就传啥
    source="weixin:o9cq809U",         # 写进 history.json 的来源标签
)
```

`route_inbound` 是 `async`——内部可能 `await ensure_daemon_alive`。其余都是
同步文件 I/O。

## RouteOutcome 契约

```python
@dataclass
class RouteOutcome:
    reply:   Optional[str] = None   # 把这段文本回给用户，到此为止
    routed:  bool          = False  # 消息已写入会话 inbox
    alias:   Optional[str] = None   # 涉及哪个会话
```

`reply` 和 `routed` 互斥——只有一个会带真实信号。渠道按结果分流：

| 结果         | 渠道动作                                                              |
|--------------|----------------------------------------------------------------------|
| `reply` 非空 | 通过渠道的发送通道把文本发给用户；`alias` 不参与本次投递路由           |
| `routed=True`| 消息已交给后端；做渠道自己的 UI（如对方正在输入气泡）                  |
| 两者都空     | 不应该发生；真发生就记日志当静默失败处理                                |

`alias` 是信息字段。渠道可以拿它更新自己的 per-alias 簿记——比如微信会把
`alias → peer_id` 落到 `weixin_state/alias_peer.json`，让 outbox watcher
在 web_server 重启后还能把 Claude 的回复送到对的人。

## 纯数字入站：一次性菜单选择（one-shot）

**铁律：控制选项与聊天不混着发；一个菜单只在"刚弹出那一回合"认数字。**

桥里有两种会"用数字选择"的菜单，但它们**绝不常驻**——只在被展示的那一刻
arm 一个一次性令牌，**下一条入站消息消费掉它，无论这条是不是数字**：

| 菜单来源 | arm 方式 | 数字命中后的动作 |
|---|---|---|
| `/proj` 项目列表（桥控制面） | `_cmd_proj` 写 `proj_choices`（含项目数据） | 见下"两级菜单"：不再直接起会话，而是弹会话菜单 |
| resume 会话列表（桥控制面，第二级） | 选完项目后写 `resume_choices`（含该项目 session 列表） | `_cmd_pick_resume(n)` 写 `RESUME:<id>` 触发 daemon resume 重启 |
| child claude TUI 菜单（后端） | daemon 检测到中继块"像菜单"时写 `control_mode_path == "menu"` | 写 `control_path`，回灌成 PTY 控制序列 |

**两级菜单（2026-07-21，resume 默认化）**：`/proj` 的语义从"选了就起/切会话"翻转为
"选了就进入 **resume 该项目历史会话**"。第一级选项目后**不直接起会话**，而是 arm 第二级
令牌 `resume_choices` 并弹出「该项目最近 5 个会话」菜单；用户再回一个
数字选中会话，`_cmd_pick_resume(n)` 把 `RESUME:<session-id>` 写进 `control_path`，触发
claude_channel daemon 的 resume 重启（kill 旧 child → 带 `--resume` 重 spawn，契约见
`后端设计.md` "resume 控制通路"）。

- **该项目无历史会话** → 跳过第二级，直接开新白纸会话（退化，等价旧 `/proj` 行为）。
- **`/new` 是"开新白纸"的显式出口**：单级（选项目直接起全新会话），不 resume、不弹第二级。
  想接上次 → `/proj`（默认 resume）；想从头 → `/new`。
- **两级都遵守 one-shot 铁律**：第一级选项目消费 `proj_choices` 同时 arm `resume_choices`；
  第二级选会话消费 `resume_choices`。任一级下一条非数字入站都解除对应令牌（控制不粘连）。
- **仅 claude_channel 支持 resume 重启**（daemon 认 `RESUME:` 前缀）。其它 backend
  选完项目/会话仍走各自的起会话路径；`RESUME:` 前缀对 claude_code 的 `control_path`
  语义无影响（那边只写纯数字）。

**会话列表与摘要来源（2026-07-22，对齐原生 `claude --resume`）**：第二级菜单的
会话集合与摘要**都以 `~/.claude/history.jsonl` 为权威源**（原生 resume 也读它），
不再扫 transcript 首条 user 消息（那样 `/recall`→`继续` 等固定开场的会话摘要会全撞）。
规则（实现见 `core/resume_scan.py`）：

- **枚举**：读 `history.jsonl`（每行 `{display, timestamp, project, sessionId}`，
  `display` = 用户敲的原话），按 `project == 该项目cwd` 过滤、按 `sessionId` 归组。
- **只列可 resume 的会话**：过滤出 `~/.claude/projects/<sanitized-cwd>/` 下**还存在
  同名 `<sessionId>.jsonl`** transcript 的会话——transcript 已被清理的死会话选了会
  `--resume` 失败，直接不显示（与原生一致）。history 里的会话数 > 可 resume 数是常态
  （老 transcript 会滚动清理）。
- **排序 + 取数**：按会话最新 `timestamp` 倒序，取最近 `RESUME_MENU_LIMIT`（=5）个。
- **每条摘要**：该会话在 history 里**最近 3 条 `display`** 拼一行（如
  `继续 · 提交 · 应该resume之前会话`），比单取一句更能让人认出"上次聊到哪"。
- **session_id**：直接用 history 的 `sessionId`（已验证 == transcript 文件名，
  可直接喂 `--resume`）。

`route_inbound` 在**最顶端**消费令牌：

1. 读并删除 `control_mode_path(alias)`（pty arm，一次性）；读 `proj_choices`
   是否 active（proj arm）。
2. 本条是纯数字 `\d+` 且某个菜单 arm 着 → 当选择，按来源分派（**proj 优先于
   pty**：两个同时 arm 时项目切换赢，pty 令牌也已被删除不残留）。
3. 否则（不是数字，或没有 arm）→ 令牌已被清掉，数字**落 inbox 当普通聊天**
   发给 child claude。proj arm 若没被数字消费，这一步也显式清掉（非数字入站
   即解除）。

**故意的副作用**：`/proj → 你好 → 2` 这种流程，`你好` 已经消费掉"菜单那一
回合"，`2` 会当聊天而不是切项目。只有 `/proj → 2`（紧接着）才切。这是
"控制与聊天不混着发"的直接代价，是要的行为，不是 bug。

**daemon 侧"像菜单"判据**（`_looks_like_menu`，工程 heuristic，可调）：中继块
含选择光标 `❯`，或有 ≥2 行匹配 `^\s*[❯>]?\s*\d+[.):、]\s+\S`（编号选项）。
普通散文回复不命中 → 纯文本中继、不 arm、不追加 `8/9/digits` 控制脚注。
`8`(auto wake)/`9`(ESC) 这类控制位同样只在 arm 着的那一回合生效。

**例外：`/stop` 命令随时可用**。one-shot 铁律限制的是**裸数字**（防止聊天
里的数字被吞成控制序列）；`/stop` 是显式 slash 命令，语义无歧义，任何时候
发都会向当前 daemon-managed 会话的 `control_path` 写 `"9"`（daemon 既有
逻辑：发 ESC 给 child claude 中断正在执行的任务，回执 `sent ESC`）。
bridge-owned 会话（claude 跑在用户自己终端里）远程中断不了，回提示让用户
去终端按 ESC。

**ESC 后必须立刻重唤循环**：ESC 打断当前回合后 child claude 停在 TUI
提示符，不会自己再调 `wait_for_message`——循环是死的，而 daemon 的
`loop_active` 标志不会被 ESC 清掉、bridge 的 busy 心跳还会继续续租
marker（最多 30 分钟），全系统误以为在服务，消息落 inbox 没人取 =
永久失联（07-17 上线当天发现）。所以 daemon 对 `"9"` 的处理是三步：
`send_escape()` → `loop_active = False` → 停 ~1s 后 `wake_loop()` 重新
键入 trigger，child claude 重进等待循环（PTY 打出 `loop active` 后
标志自动恢复）。被中断的任务保持中断，会话保持在线。

> 历史包袱：在此规则前 `control_mode` 是**全程常开**的（daemon 启动/启动后/
> 每次中继块都 set True，仅退出时 False），导致 `/proj` 120s 窗口外任何数字
> 都被当 PTY 控制吞掉。one-shot 改造同时废掉了这个常开态与 120s 时间窗
> （`proj_choices.expires_at` 降级为"无人接话时的兜底 TTL"，主语义是一次性）。

## route_inbound 内部决策顺序

1. **是否 slash 命令** — `commands.is_command(text)` 对单 `/` 命令、或者
   `/proj` 选号窗口内的纯数字返回 True。命中就跑 `commands.handle_command`
   返回 reply，不动后端。（数字 arm 的消费见上节"纯数字入站"，发生在本步之前。）

   命令里有一类"**元命令**"——不直接操作当前会话，而是改"之后建会话时的
   参数"。目前只有 `/backend [<name>]`：看/切默认 backend，落
   `chat_sessions/_default_backend.txt`，之后 `/proj N` / `/proj 0` 建会话
   时读它。详见 `后端设计.md` "默认 backend 契约"段。

2. **passthrough (`//xxx`)** — `commands.strip_passthrough_prefix` 去掉一层
   `/`，让 agent 看到的是 `/xxx`，剩下的文本走普通消息路径。

3. **没有当前会话** — `sx.get_current()` 返 None。**直接弹 `/proj` 项目菜单**
   （等价于用户自己敲了 `/proj`），让他就地选项目；当前这条消息丢弃。
   复用 `commands.handle_command("/proj")` 生成，菜单格式/分页/arm 全一致。
   - 改动缘由（2026-07-20）：原先回"请去 dashboard 建一个"，但微信用户不该被
     赶去浏览器——零会话时收到普通消息，最自然的响应就是把项目菜单推给他。
   - 语义翻转后（2026-07-21）：`/proj` 现在是"进入 resume 该项目历史会话"的默认
     路径（见上"两级菜单"）。所以零会话随便打字 → 弹项目菜单 → 选项目 → 弹会话
     菜单 → 选会话 → resume 接回上下文。"想接着上次聊"就走这条默认路径；想开全新
     白纸另发 `/new`。
   - 边界：只对**普通消息**触发（命令/数字菜单选择在第 1 步已分派掉，到不了这里）。
   - 触发前提是 `_current.txt` 为空。删空 chat_sessions / 首次启动 / 清理后都会
     进这个状态，属正常，不是错误。

4. **idle gate** — `sx._last_active(alias)`（即 inbox/outbox/history 三个
   文件 mtime 的最大值）超过 `IDLE_THRESHOLD_SECS` 就触发，返回 `/proj`
   列表让用户显式选继续或新开。**不动 daemon**，当前消息丢弃。阈值在
   `core.router.IDLE_THRESHOLD_SECS`（目前 2 小时）。

5. **执行端可服务性** — 先看会话属于哪类（见下节"终端 chats-loop 会话"）：
   - **bridge-owned**（daemon 不活、但 `meta.bridge_pid` 活着，即某个终端
     claude 挂着 cca-msg MCP）：**绝不在活 bridge 上叠 daemon**——会出现
     两个进程抢读同一个 inbox。此时看 marker 文件
     `~/.claude/.chats-loop-active-<alias>`：在（chats-loop 循环真在收件）
     → 直接投递；不在（只是挂着 MCP、没跑循环，消息没人接）→ 返回
     `/proj` 菜单让用户显式选会话/项目，当前消息丢弃。
   - **其余（daemon-managed / 全死）** — 调 `spawn.ensure_daemon_alive(alias)`：
     daemon 活着直接过，死了拉起。返 False 就把"agent 拉起失败"作为 reply
     抛回去。spawn 成功的话，**会顺带启动一个就绪通知后台任务**（见下节
     "就绪通知"）。

6. **交给后端** — 把 `text` **追加**进 `inbox_path(alias)`（换行分隔；
   后端处理中时多条入站排队积压，下轮 wait 一次取走，不覆盖丢失），
   追加一条 history，返 `routed=True`。

## 终端 chats-loop 会话（bridge-owned）

用户在任意终端手开 claude 并触发 `/chats-loop` 时，没有 daemon——只有
mcp_bridge 进程直接伺候 inbox/outbox。这类会话的契约：

- **注册（被动可发现）**：mcp_bridge 启动和 `relay_init` 时写自己 session
  目录的 `meta.json`（`bridge_pid` + `bridge_create_time`），让
  `/proj`、`/list`、dashboard 能列出它、在线判定能看到它。
- **绝不写 `_current.txt`**。成为"当前会话"只能靠用户显式选择（`/proj`
  选号、`/use`、dashboard、菜单一次性选号）。教训（2026-07-16）：任何
  手开的 claude 都会自动拉起 bridge，如果 bridge 启动即抢路由指针，等于
  随手开个窗口就劫持了所有微信入站——消息落进没人消费的 inbox，全网静默。
- **在线 = `bridge_pid` 活 && marker 新鲜**。bridge 进程活着只说明 MCP server
  挂着；真的在收件必须看 `~/.claude/.chats-loop-active-<alias>` 的
  **mtime 是否在 TTL（`paths.LOOP_MARKER_TTL_SECS`，180s）内**。marker
  是租约不是开关：mcp_bridge 在**等待中和处理中**都每 ~5s touch（处理中
  由 bridge 后台线程代打，上限 30 分钟，见"就绪通知/信号源"节）、
  `send_chat_response` 时也 touch；循环停了（哪怕进程还活着、哪怕
  文件还在）mtime 不再刷新，TTL 过后自动判离线。**只看"文件存在"是不够
  的**：进程被硬杀时 atexit 不跑，残留 marker 会造成"循环停了还显示在线"
  的假阳性（2026-07-16 实测踩到）。判定一律用 `paths.loop_marker_fresh()`。
  `list_sessions` / `list_projects` 的 online 按
  `daemon 活 || (bridge 活 && marker 新鲜)` 计算。
- **PID 复用防护**：`_reconcile_meta_liveness` 对 `bridge_pid` 跟
  `daemon_pid` 一样用 `create_time` 二次校验，死了/被复用就清成 null。
- **同 cwd 多个 session 时**，`list_projects` 的项目→alias 归属**在线者
  优先**（都在线/都离线才是后扫到的赢），保证菜单选中的是真能接话的那个。

## 就绪通知

任何由 bridge 主动拉起 daemon 的场景（router 的 ensure_daemon_alive、
autospawn worker 处理 `/proj N` 或 `0` 写进的队列项），spawn **成功**后必须
启动一个后台 watch 任务，监听这个 alias 的 chats-loop skill 何时真激活，
然后向用户广播就绪/失败。

### 信号源

mcp_bridge 在 child claude 第一次进入 `wait_for_message` 时 touch 一个
marker 文件：

```
~/.claude/.chats-loop-active-<alias>
```

这个文件**出现** = skill 已激活（watch_ready 等的就是它的出现）。但"当前
是否仍在收件"要看它的 **mtime 新鲜度**，不是存在性——mcp_bridge 在两个
阶段持续心跳（每 ~5s touch）：

1. **等待中**：`wait_for_message` 阻塞轮询期间；
2. **处理中**：`wait_for_message` 返回真消息起、到下一次 `wait_for_message`
   为止（模型正在执行任务，bridge 进程用后台线程代打心跳；上限
   `BUSY_MAX_SECS` 30 分钟，超过视为模型挂死，停止续约）。

`send_chat_response` 时也 touch。租约 TTL 见 `paths.LOOP_MARKER_TTL_SECS`。
"处理中也算在服务"是关键语义：模型埋头干活几分钟时，后续入站消息必须
正常落 inbox 排队（模型下轮 wait 取走），而不是被误判"没人收件"弹菜单
（2026-07-17 实测踩过：执行超 180s 后追发消息被 /proj 菜单吞掉）。

文件不出现 = skill 没起来（trust 弹窗卡住、READY_MARKERS 不匹配、trigger
没进 TUI…）；文件在但 mtime 陈旧 = 循环已停 / 模型挂死超 30 分钟 / 进程
被硬杀后的残留。

### 行为契约

`spawn.watch_ready(alias)` 是一个 `async` 函数，由调用 spawn 的位置
`asyncio.create_task` 起：

| 情况 | 动作 |
|---|---|
| `READY_NOTIFY_TIMEOUT` 内 marker 出现 | 写 outbox：`✅ 会话 <alias> 已就绪，发消息试试` |
| `READY_NOTIFY_TIMEOUT` 超时 marker 仍不存在 | 写 outbox：`⚠️ 会话 <alias> 拉起超时，可能 child claude 卡在了某个弹窗。看 chat_sessions/<alias>/pty.log` |
| 期间 daemon 进程死了 | 写 outbox：`⚠️ 会话 <alias> 启动后异常退出，看 daemon.log` |

`READY_NOTIFY_TIMEOUT` 在 `core.spawn` 里（默认 60 秒——含 trust 弹窗 +
TUI 渲染 + `/chats-loop` 触发 + skill 初始化 + 第一次 wait_for_message
的累计时间）。

### 为什么是写 outbox，而不是渠道单独发送

outbox 是后端给前端发消息的现有通道——所有渠道都已经接好 outbox_watcher。
通过 outbox 发就绪通知意味着：(1) 渠道一行代码不用动；(2) 用户在哪个渠道
聊就在哪个渠道收到，自动多渠道适配；(3) 同样的通知在 web UI `/poll`
也能拿到，无差别。

唯一注意：outbox 是"最新一条待推"的单槽，watch_ready 写完通知后如果
child claude 立刻又回了别的，可能盖掉就绪通知——所以**就绪通知必须在
child claude 收到第一条消息之前发出去**，靠 marker 文件的 touch 时机
天然保证。

### 不发就绪通知的场景

- 用户手动起的 daemon（`python -m chats_control_agents.backends.claude_code.daemon`）
  ——他自己知道在等，不需要桥通知。
- web dashboard 的"开始新会话"按钮——HTTP 请求本身会同步等 ready 后才返
  回，UI 已经能反映状态。

## 出站推送契约：outbox_watcher 只推当前会话（2026-07-21）

入站决定"消息进哪个会话"，出站决定"哪个会话的 outbox 推给用户"。两者必须
用同一个视角：**微信是单窗口，`_current` 单选，所以 outbox_watcher 也只推
`sx.get_current()` 那一个 alias 的 outbox**。

**铁律：每轮只看 current。** watcher 主循环每 0.5s 只检查当前会话的
`outbox.txt`，非当前会话的 outbox 有新内容也**不推微信**（web UI `/poll`
不受影响——那是另一条路，浏览器可以并行看多个会话）。

**切走的会话不丢消息，但也不打扰你。** 用户视角：我在看哪个会话，就只收哪个。

- 会话 A 是 current 时你发消息 → A 后台算（比如 30s）→ 这期间你切到 B →
  A 算完把回复写进 A 的 `outbox.txt`。此刻 current=B，A 的回复**不推**。
- 那条回复**静静躺在 A 的 outbox 里**，没被标记已送达（`_outbox_seen` 里没有
  它）。等你**切回 A**，watcher 视角转到 A，发现 A 的 outbox 有未送达内容
  → 自然推给你。**这不是"补推逻辑"，是"只推 current"规则的自然结果**：
  没送达的一直在 outbox 等着，视角回到它就见到。

**`_outbox_seen` 的语义 = "已成功送达"，不是"已存在"。** 只在 `send_text`
成功后才写（成功才算送达）。切换会话时**不做任何预标记 / 不预标已见**——
否则会把"还没发给用户的真实回复"当成已见吞掉，那是丢消息。切换纯靠"每轮
只看 current"自然生效。

**命令的即时回复不经这条路。** `/proj`、idle-gate、零会话菜单这些走
`route_inbound` 的 **reply 分支**，由渠道直接 `send_text` 回执（见上"决策
顺序"），跟 outbox_watcher 无关。outbox_watcher 只管**后端会话产生的回复**。

**改动缘由**：原 watcher 是 `for sess in list_sessions()` 的**多会话广播**
——任何登记过 peer 的会话吐内容都推给那个 peer，与"全局单选 `_current`"
自相矛盾。2026-07-21 实测：切到一个会话后，它 outbox 里的内容被推给用户，
即使那不是用户当前关心的会话。改为只推 current 后，出站视角和入站的
`_current` 单选对齐。

> 不在这轮的：outbox 里的 **TUI 启动噪声**（claude_code 会话启动 chats-loop
> 时把过程刷屏写进 outbox）本身该不该产生——那是 backend 层的独立问题，
> 与"推哪个会话"正交，留待后续。停在一个正在刷 TUI 的 current 会话上仍会
> 看到噪声。

## 为什么不让渠道自己做

这套步骤放进核心，新加渠道就只剩一件事：解析协议帧 → 调 `route_inbound`
→ 按 outcome 分流。2 小时 idle gate、`/proj` 重发、按需拉 daemon、no-session
提示——只要新渠道接进来这套立刻可用。

## 不在 router 里的事

- 渠道协议（HTTP、long-poll、auth）—— 渠道自己负责。
- 渠道的持久化（`alias_peer.json`、peer 上下文 token）—— 渠道自己负责，因为
  peer 身份本就是渠道概念。
- 后端协议（`inbox.txt` 怎么进 child claude、`outbox.txt` 怎么生成）—— 后端
  自己负责。
- UI 装饰（输入气泡、已读回执、消息编辑）—— 渠道自己负责，在拿到
  `routed=True` 时触发。

## idle gate 设计动机

旧行为：任何入站都会静默拉起当前会话的 daemon。隔几天没说话，"新"消息
落到陈年 alias 里，用户也没法干净切换。idle gate 强制在 2h 节点显式让用户
选——紧凑对话不会触发，第二天回来才出现，是正好的位置。

阈值用秒不是分钟，方便以后扩（按渠道差异化、按时间段调）。
